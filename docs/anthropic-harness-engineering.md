# Anthropic 하네스 엔지니어링 핵심 정리

> "하네스(harness)"란 LLM(예: Claude)을 감싸는 오케스트레이션 레이어를 의미한다.
> 모델 자체가 아니라, 모델에게 **무엇을 주고**, **무엇을 받고**, **어떻게 행동을 제어할지**를 설계하는 영역이다.
> Anthropic은 Claude Code를 통해 이 하네스 엔지니어링의 모범 사례를 제시한다.

---

## 1. 핵심 철학: 모델에게 좋은 환경을 만들어 줘라

Anthropic이 반복해서 강조하는 원칙은 다음 세 가지로 요약된다.

1. **좋은 도구(Good Tools)** — 모델이 작업을 끝낼 수 있는 정확한 도구를 제공하라.
2. **좋은 컨텍스트(Good Context)** — 모델이 결정을 내리는 데 필요한 정보를 주되, 노이즈는 줄여라.
3. **좋은 피드백 루프(Good Feedback Loops)** — 도구 실행 결과를 명확히 돌려줘서 모델이 자기 행동을 교정할 수 있게 하라.

이 세 가지를 만족하지 못하면 모델 성능이 아무리 좋아도 에이전트는 실패한다.

---

## 2. 컨텍스트 관리 (Context Management)

컨텍스트 윈도우는 유한한 자원이다. 하네스 엔지니어링의 절반은 컨텍스트 관리다.

- **자동 컴팩션(Compaction)**: 컨텍스트 한계에 다다르면 이전 메시지를 자동으로 요약·압축한다. 단순 잘라내기(truncation)가 아니라, 의미를 보존하는 압축을 우선한다.
- **서브에이전트(Subagent)로 컨텍스트 보호**: 대량의 검색·읽기 작업은 서브에이전트에 위임하면, 메인 컨텍스트에는 결과 요약만 들어온다.
- **중요 정보의 명시적 보존**: 도구 결과가 나중에 사라질 수 있으므로, 모델은 중요한 정보를 응답 본문에 옮겨 적는 습관을 가져야 한다.

---

## 3. 도구 설계 (Tool Design)

도구가 너무 많거나 모호하면 모델은 혼란에 빠진다.

- **전용 도구를 우선하라**: `Read`, `Edit`, `Write` 같은 전용 도구가 있으면 셸 명령(`cat`, `sed`)보다 항상 그것을 써야 한다. 의도가 명확해지고, 권한 시스템·UI 표시·로그 모두에서 이점이 있다.
- **도구는 적게, 명확하게**: 도구 수가 늘어날수록 토큰 비용과 선택 오류가 함께 증가한다.
- **병렬 호출 가능하게 설계**: 독립적인 작업은 한 응답에서 동시에 호출하도록 구조화한다.
- **도구의 부작용은 명시적으로**: 파괴적·비가역적 동작은 별도 도구로 분리하고, 권한 시스템과 결합한다.

---

## 4. 권한과 안전 (Permissions & Safety)

Anthropic 하네스의 큰 차별점은 **단계적 권한 시스템**이다.

- **퍼미션 모드**: 사용자가 자동 허용 범위를 결정한다.
- **`allow` / `deny` 목록**: `settings.json`에서 도구별로 세밀히 통제한다.
- **거부 시 재시도 금지**: 사용자가 거부한 동일한 호출을 그대로 재시도하지 않고, 의도를 다시 해석한다.
- **블래스트 반경(Blast Radius) 평가**: 되돌리기 어렵거나 공유 자원에 영향을 주는 작업은 사전 확인.
  - 예: `git push --force`, `rm -rf`, DB 스키마 변경, 외부 메시지 송신 등.

---

## 5. 훅(Hooks) — 자동화의 결정적 도구

훅은 특정 이벤트(`PreToolUse`, `PostToolUse`, `SessionStart`, `Stop` 등)에 셸 명령을 실행하는 메커니즘이다.

- "X 할 때마다 Y 하라"는 자동화는 **메모리/지시문이 아니라 훅으로** 구현해야 한다. 메모리는 모델의 의지에 의존하지만, 훅은 하네스가 결정론적으로 실행한다.
- 일반적인 활용: 포맷터·린터 자동 실행, 커밋 검증, 세션 시작 시 환경 점검, 종료 시 알림 발송 등.

---

## 6. 서브에이전트와 작업 분할

복잡한 작업은 단일 컨텍스트에서 처리하기보다 분할이 유리하다.

- **탐색형(Explore) 서브에이전트**: 코드 검색·파일 위치 확인 등 읽기 전용 탐사.
- **계획형(Plan) 서브에이전트**: 구현 전략 설계.
- **범용(General) 서브에이전트**: 다단계 작업 위임.
- **병렬 실행**: 의존성 없는 작업은 한 메시지에서 동시에 시작한다.

서브에이전트는 메인 컨텍스트의 토큰을 절약하고, 결과 요약만 메인으로 돌려주는 것이 핵심 가치다.

---

## 7. 스킬(Skills)과 슬래시 명령(Slash Commands)

- **스킬**: 도메인 지식이나 절차를 모듈화한 재사용 가능한 지시문 묶음. 필요할 때 호출한다.
- **슬래시 명령**: 사용자가 명시적으로 트리거하는 단축 명령. 예: `/review`, `/init`, `/security-review`.
- 둘 다 "프롬프트 비대화"를 막으면서 기능을 확장하는 수단이다.

---

## 8. MCP(Model Context Protocol) 서버

외부 도구와 서비스를 모델에게 노출하는 표준화된 인터페이스.

- GitHub, Atlassian, 카카오, Microsoft 365 등 다양한 MCP 서버를 통해 모델이 직접 외부 시스템과 상호작용한다.
- 권한·도구 스키마·인증을 서버 측에서 캡슐화하므로, 하네스는 깔끔하게 유지된다.

---

## 9. 메모리: `CLAUDE.md`

프로젝트 특화 지시문은 코드와 함께 버전 관리되는 `CLAUDE.md`에 둔다.

- 코딩 컨벤션, 자주 쓰는 명령, 프로젝트 컨텍스트, 금지 사항 등을 적는다.
- 이 파일은 세션 시작 시 자동으로 로드되어 모델의 기본 프롬프트가 된다.
- 단, **자동화가 필요한 규칙은 메모리 대신 훅으로** 처리해야 한다는 점을 잊지 말 것.

---

## 10. 설정 계층 (Settings Hierarchy)

`settings.json`은 다음 우선순위로 병합된다.

1. **Local** (`.claude/settings.local.json`) — 개인용, 커밋되지 않음.
2. **Project** (`.claude/settings.json`) — 팀 공용, 레포에 커밋.
3. **User** (`~/.claude/settings.json`) — 머신 전역.

권한·환경변수·훅·MCP 서버를 모두 이 파일에서 제어하며, 계층을 잘 이해해야 팀과 개인 설정이 충돌 없이 공존한다.

---

## 11. 출력 톤과 작업 행동의 표준화

하네스는 모델의 **결과물 자체의 품질**도 통제한다.

- 불필요한 주석·과잉 추상화·미래 가정 코드 금지.
- 외부 경계(사용자 입력, 외부 API)에서만 검증하고, 내부 신뢰는 가정.
- UI 변경은 실제 브라우저 확인 후 보고. 못 했으면 솔직히 말할 것.
- 위험한 작업(force push, hard reset, 외부 메시지 송신)은 항상 사전 확인.

---

## 12. 실전 체크리스트

새로운 에이전트 하네스를 설계할 때:

- [ ] 모델이 끝낼 수 있는 **최소한의 도구 집합**이 정의되어 있는가?
- [ ] 도구의 **결과 형식**이 모델에게 명확한가?
- [ ] **컨텍스트 압축/요약** 전략이 있는가?
- [ ] **권한 시스템**으로 위험 작업이 제어되는가?
- [ ] 반복적·결정론적 자동화는 **훅**으로 분리되었는가?
- [ ] 큰 탐사·다단계 작업을 위한 **서브에이전트**가 마련되었는가?
- [ ] 프로젝트 지식은 **메모리(`CLAUDE.md`)** 로 정리되었는가?
- [ ] **블래스트 반경**이 큰 작업은 사전 확인 흐름이 있는가?
