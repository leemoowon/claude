# 서브에이전트 구성 전략 & 에이전트팀 활용 가이드

> 대상: 이 워크스페이스(mwlee@telechips.com)의 실제 업무를 Claude Code로 자동화·분담할 때
> "무엇을, 어떤 단위로, 어떤 프리미티브로" 나눌지 결정하기 위한 가이드.
> 한 줄 결론: **지금 업무는 서브에이전트 + 스킬로 충분하다. 에이전트팀은 "아직"이며, 도입 트리거가 충족될 때만 쓴다.**

---

## 0. 30초 요약

- Claude Code의 자동화 수단은 6가지다: **Hook → Command → Skill → Subagent → Agent Team → Workflow**.
  뒤로 갈수록 강력하지만 비싸고 복잡하다. **모든 걸 서브에이전트로 만들지 마라.**
- 서브에이전트는 **"프로젝트"가 아니라 "역할/관심사"** 단위로 나눈다. (리뷰어, 탐색가, 리서처…)
- 현재 보유 자산(`code-reviewer`, `doc-reviewer`, `daily-report`, `deep-research`, `/new-project`, hooks)은
  이미 업무에 잘 맞다. **큰 공백 없음.**
- 에이전트팀은 **토큰 ~7배·실험적**. 지금 일상 업무엔 과하다. 진짜 멀티레이어/대형 작업 + "에이전트 간 토론이
  실익"일 때만 켠다. → 본 문서 [섹션 D](#d-에이전트팀--판정과-도입-트리거), [섹션 E](#e-활용-시나리오) 참고.

## A. 6가지 오케스트레이션 프리미티브 — "무엇을 언제 쓰나"

| 신호(이런 일이면) | 프리미티브 | 핵심 이유 | 이 워크스페이스의 예 |
|---|---|---|---|
| 자동·결정론적 트리거(세션 시작/종료 등) | **Hook** | 사람 판단 불필요, 코드로 고정 실행 | `session-start-status.sh`, `auto-push.sh` |
| 짧고 정형화된 반복 프롬프트(인자를 받음) | **Command** | 빠른 호출, 컨텍스트 격리 불필요 | `/new-project` |
| 다단계 절차 + 스크립트/도메인지식 묶음 | **Skill** | 절차·지식을 패키징, **메인 세션이 직접 수행** | `daily-report`, `deep-research` |
| 컨텍스트 격리가 필요한 **위임형 단일 작업**(리뷰/탐색/리서치) | **Subagent** | 자체 컨텍스트에서 일하고 **요약만 반환** → 메인 오염 방지, `description`으로 자동 위임 | `code-reviewer`, `doc-reviewer` |
| 여러 에이전트가 **병렬로 일하며 서로 토론·조정**해야 가치가 나는 작업 | **Agent Team** | 피어 간 `SendMessage` + 공유 태스크리스트. 단 **~7배 비용·실험적** | (아직 없음) |
| 수십~수백 파일 **대규모 정형 처리**(감사/마이그레이션) | **Workflow(JS)** | 스크립트가 계획을 보유, 대량 스케일·교차검증 | (아직 없음) |

### 결정 트리 (위에서부터 해당하는 첫 항목을 고른다)

1. **자동/결정론적**으로 매번 똑같이 일어나야 하나? → **Hook**
2. **한 줄짜리 반복 프롬프트**(인자만 바뀜)인가? → **Command**
3. **"절차서 + 도구 호출 묶음"** 인가? → **Skill**
4. **"딴 데서 조용히 해와서 결론만 줘"** (메인 대화를 더럽히기 싫다)인가? → **Subagent**
5. **"여러 명이 동시에 + 서로 의견을 주고받아야"** 하나? → **Agent Team**
6. **"파일 100개를 일괄로"** 인가? → **Workflow**

> 흔한 실수: 4번이면 충분한데 5번(팀)을 쓰는 것. 서브에이전트도 **병렬 실행**이 된다.
> 팀이 서브에이전트와 다른 단 하나의 본질은 **"에이전트끼리 직접 대화(SendMessage)"** 다.

## B. 서브에이전트 분할 원칙 — "프로젝트가 아니라 역할로 나눠라"

서브에이전트를 만들 때 지킬 4원칙:

1. **단일 책임 (Single Responsibility)**
   - 에이전트 1개 = 관심사 1개. (리뷰 / 탐색 / 리서치 / PR요약 …)
   - ❌ `android-project-agent`, `sdk-project-agent` 처럼 **프로젝트별로** 나누지 말 것.
     같은 "코드 리뷰" 역할이면 프로젝트가 달라도 `code-reviewer` 하나로 충분하다.
   - ✅ `code-reviewer`(코드), `doc-reviewer`(산문)처럼 **다루는 대상/관점**으로 나눈다.

2. **도구 최소화 (Least Privilege)**
   - `tools` 화이트리스트로 꼭 필요한 것만. 예: `code-reviewer`는 `Read, Bash, Grep, Glob`만 — **쓰기 도구 없음**.
   - 리뷰어가 코드를 고치지 못하게 막으면, 검토와 수정의 경계가 분명해진다.

3. **날카로운 `description` (자동 위임의 핵심)**
   - 메인 Claude는 각 서브에이전트의 `description`을 읽고 **언제 호출할지** 자동 판단한다.
   - "무엇을 하는지"보다 **"언제 부르는지"** 를 트리거 문구로 적어라.
     예) `doc-reviewer`: "회의록·리포트·README 같은 **산문**을 리뷰할 때. 사용자가 '이 문서 봐줘'라고 하면 PROACTIVELY 호출."

4. **모델 차등 (Cost-aware)**
   - 단순 위임(검색/요약)은 `model: sonnet` 또는 `haiku`로 비용 절감.
   - 깊은 추론이 필요한 메인 작업만 `opus`. 서브에이전트 대부분은 메인보다 가벼워도 된다.

### 서브에이전트 vs 스킬, 헷갈릴 때
- **결과만 받으면 되고, 과정이 길어 메인 대화를 더럽힐 것 같다** → Subagent (격리된 컨텍스트, 요약 반환)
- **내가(메인이) 직접 단계를 밟아야 하고, 그 절차/지식을 재사용하고 싶다** → Skill
- 예: `daily-report`는 메인이 직접 여러 MCP를 호출해 형식을 맞추는 **절차**라 스킬이 맞다.
  반대로 "이 50페이지 로그에서 원인만 찾아와"는 격리가 이득이라 **서브에이전트**가 맞다.

## C. 내 업무 단위 → 프리미티브 매핑 (권장 로스터)

**굵게 = 이미 보유 / (신규 후보) = 지금 만들지 않지만 필요해지면 추가 고려**

| 업무 단위 | 권장 프리미티브 | 비고 |
|---|---|---|
| 하네스/제품 코드 작성 | 메인 세션 | 대화형·창의형 작업, 격리 부적합 |
| 코드 리뷰 | **Subagent `code-reviewer`** | 유지 — 버그/보안/가독성, 커밋·PR 전 |
| 문서/회의록 리뷰 | **Subagent `doc-reviewer`** | 유지 — 산문 전용, 구조/사실/톤 |
| 코드베이스 탐색·검색 | 내장 **Explore** 서브에이전트 | fan-out 검색 후 결론만 받기 |
| 심층 리서치 | **Skill `deep-research`** | 유지 — 다중 소스·검증·인용 |
| 일일 보고(멀티소스 집계) | **Skill `daily-report`** | 유지 — Jira/Confluence/Teams/Outlook |
| 새 프로젝트 스캐폴딩 | **Command `/new-project`** | 유지 |
| 커밋/푸시·세션 상태 | **Hook** | 유지 — auto-push, session-start |
| PR 본문/리뷰 요약 작성 | (신규 후보) Subagent `pr-writer` | diff→PR 설명/요약을 반복 위임하게 되면 |
| Jira/Confluence 정리·발행 | (신규 후보) Skill | MCP 호출 절차가 정형화되면 패키징 |

**결론: 현재 셋업은 이미 업무에 잘 맞춰져 있고 큰 공백이 없다.** 신규 후보 2개는 "그 일을 반복적으로
손으로 위임하고 있다"는 신호가 나타날 때 추가하면 된다. 미리 만들지 않는다(YAGNI).

## D. 에이전트팀 — 판정과 도입 트리거

### 판정: 지금은 **불필요**(과하다)

근거:
- 현재 업무는 **순차적·단일 컨텍스트**다(리뷰 1건, 문서 1건, 보고 집계 1회…).
- `projects/`·`experiments/`가 비어 있어 **병렬화할 대형 코드베이스가 아직 없다.**
- 에이전트팀은 **실험적**(`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`)이고, **토큰 ~7배** +
  조정(coordination) 오버헤드가 든다.

### 도입 트리거 (아래 **둘 다** 충족될 때만)

1. **규모/구조**: 진짜 멀티레이어거나 대형인 작업이 생겼다.
   예) 드라이버 + 커널 통합 + 테스트 + 문서가 한 번에 얽힌 50+ 파일 변경.
2. **상호작용 이득**: 단순 병렬이 아니라 **에이전트끼리 발견을 주고받고 교차검증**하는 것이 실익이다.

### 서브에이전트 vs 에이전트팀 — 한 줄 차이

> **서브에이전트는 서로 대화하지 못하고 각자 메인에만 보고한다.**
> 에이전트팀은 팀원끼리 `SendMessage`로 직접 대화하고 공유 태스크리스트로 일을 나눈다.
> → **"교차검증·토론 그 자체가 가치"인 경우에만 팀**을 쓴다. 단순 분담·병렬이면 서브에이전트로 충분.

## E. 활용 시나리오

### 시나리오 1 (메인) — "대형 SDK PR 다각도 동시 리뷰" 〔임베디드/SoC 맥락〕

**상황**: 동료가 새 주변장치 드라이버를 올렸다. 50+ 파일에 드라이버 + 커널 통합 + 테스트 + 문서가 섞여 있다.
혼자 순차로 보면 느리고, 보안·성능·테스트 관점이 머릿속에서 뒤섞인다.

**팀 구성**:
- **Team Lead (메인 세션)** — PR을 영역별로 분해해 **공유 태스크리스트**를 만들고, 마지막에 종합 리뷰를 작성.
- **Teammate A — 보안 리뷰어**: 메모리 안전성(버퍼 오버런/UAF), 입력 검증, 권한 경계.
- **Teammate B — 성능 리뷰어**: ISR(인터럽트) 지연, 락 경합, 핫패스 할당.
- **Teammate C — 테스트/문서 리뷰어**: 커버리지 갭, 엣지케이스 누락, API 문서 정확성.

**팀이 빛나는 순간 (← 핵심)**:
> A가 "이 DMA 버퍼는 **ISR 컨텍스트에서도 접근된다**"는 사실을 발견한다.
> A는 이를 `SendMessage`로 **B에게 직접 전달**한다.
> B는 이 단서를 받아 "그렇다면 이 버퍼를 보호하는 스핀락이 ISR-safe(`irqsave`)인지"를
> 자기 분석에 즉시 반영한다.
> 세 리뷰어가 서로의 발견을 교차검증·반박하며 빠르게 수렴하고, Lead가 합쳐 하나의 일관된 리뷰를 낸다.

**왜 서브에이전트로는 이게 안 되나**:
> 서브에이전트 A/B/C는 서로 대화하지 못한다(각자 메인에만 보고).
> 그래서 A의 "ISR 버퍼" 단서가 B의 락 분석으로 **자동으로 흘러가지 않는다.**
> 메인이 일일이 중계해야 하고, 그 사이 맥락이 유실된다.
> **이 "에이전트 간 교차전달"이 바로 팀에만 있는 가치다.**

### 시나리오 2 (미니, 비코드) — "분기 기술 리포트 동시 집필"

여러 팀원이 섹션(시장 동향 / 자사 성과 / 리스크)별 초안을 **동시에** 쓰고, 서로의 수치·주장을
`SendMessage`로 **교차 사실검증**한 뒤 Lead가 하나의 리포트로 통합. → 단순 분담이 아니라
"숫자 충돌을 서로 잡아주는" 상호검증이 필요할 때 팀이 유리.

## F. 비용·주의·운영

- **비용**: 팀원 수에 비례해 토큰 **~7배**. 팀원은 각자 독립 컨텍스트를 유지한다.
- **규모**: **3~5명** 권장. 6명 이상은 조정 오버헤드로 수확체감.
- **모델**: 팀원은 **Sonnet**으로 충분한 경우가 많다(조정·리뷰). Lead만 무겁게.
- **정리**: 일이 끝나면 팀을 **반드시 정리**한다. 유휴 팀원도 토큰을 먹는다.
- **피하기**: 순차 의존이 많은 작업, 같은 파일 동시 편집(충돌), 단순 위임(서브에이전트가 더 쌈).
- **실험 플래그**: `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`.
- **환경 ✅ (실측)**: 이 워크스페이스의 **Claude Code on the web(원격 컨테이너) 세션**에서 플래그를 켠 결과,
  팀 도구(`TeamCreate`/`SendMessage`/`TeamDelete`)가 노출되고 **실제로 동작**함을 확인했다.
  (2026-05-30 시연: `auto-push.sh`를 보안·정책 두 팀원이 병렬 리뷰하고, 보안 팀원이 발견을 정책 팀원에게
  `SendMessage`로 직접 전달 → 정책 팀원이 이를 분석해 반박/조정.)
  - **한 가지 주의**: 이 원격 세션에서 팀원은 **일회성(one-shot)으로 실행되어 결과를 반환**하는 형태로 동작했다.
    로컬 CLI/데스크톱의 인터랙티브 세션처럼 팀원이 계속 살아 있으면서 여러 턴에 걸쳐 대화하는 경험은
    로컬에서 더 매끄럽다. 장시간·다턴 협업이 필요하면 로컬 CLI를 권장.

## 부록 1. 에이전트팀 실험 켜기 (로컬)

1. 플래그를 **개인 로컬 설정**에 추가한다(공유 `settings.json` 대신 — 팀원에게 7배 비용을 강제하지 않도록).
   `.claude/settings.local.json` (이 파일은 gitignore 됨):
   ```json
   {
     "env": { "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1" }
   }
   ```
2. 세션을 연다. **웹/원격 세션에서도 플래그가 적용되면 팀 도구가 노출된다**(본 워크스페이스에서 실측).
   단, 팀원이 여러 턴에 걸쳐 계속 살아 있는 인터랙티브 협업은 **로컬 CLI/데스크톱**이 더 매끄럽다.

## 부록 2. 바로 붙여넣는 샘플 팀 프롬프트 (시나리오 1)

로컬 CLI 세션에서 아래를 그대로 붙여넣으면 시나리오 1을 재현할 수 있다(PR 번호만 교체):

```
PR #123 을 3명으로 구성된 에이전트팀으로 리뷰해줘.
- 보안 리뷰어: 메모리 안전성(버퍼 오버런/UAF), 입력 검증, 권한 경계
- 성능 리뷰어: ISR 지연, 락 경합, 핫패스 할당
- 테스트/문서 리뷰어: 커버리지 갭, 엣지케이스, API 문서 정확성
규칙: 한 리뷰어가 다른 관점에 영향을 주는 사실(예: 어떤 버퍼가 ISR에서 접근됨)을 발견하면
SendMessage 로 해당 리뷰어에게 직접 알려라. 서로의 발견을 교차검증하고, 너(Lead)는 마지막에

```

> 팀을 켜기 싫고 상호 토론 없는 **병렬 분담**만 필요하다면, 서브에이전트 3개를 한 메시지에서 병렬
> 호출하면 된다. "병렬 분담"까지는 서브에이전트로 되고, 거기에 **피어 토론(SendMessage)**이 더해진 것이
> 진짜 에이전트팀이다.
